# 服务可用性（Uptime Kuma）配置指南

New API 首页「服务可用性」卡片展示的是**站点/API 是否在线**，数据来自外部的 Uptime Kuma 监控系统。

**注意区分：**

| | 服务可用性（Uptime Kuma） | 渠道测试（New API 自带） |
|---|---|---|
| 测什么 | 你的网站/域名/端点能不能被访问到 | 上游 AI 渠道能不能正常调用模型 |
| 入口 | 首页「服务可用性」卡片 | 渠道管理列表 |
| 配置位置 | 系统设置 → 仪表盘设置 → Uptime Kuma | 系统设置 → 运营设置 → 监控设置 |

两者互不替代，建议都配。

## 一、部署 Uptime Kuma

```bash
docker run -d --restart=always -p 3003:3001 \
  -v uptime-kuma:/app/data \
  --name uptime-kuma \
  louislam/uptime-kuma:1
```

- `-p 3003:3001`：宿主机端口按需换成未占用的，容器内固定监听 3001
- 部署前用 `lsof -i :3003` 确认端口没被占用
- 访问 `http://服务器IP:端口` 完成管理员账号初始化

## 二、添加监控项

后台点「添加监控项」，常规填法：

| 字段 | 示例 |
|---|---|
| 监控类型 | `HTTP(s)`（测真实模型连通性用 `Keyword`，见第五节） |
| 显示名称 | `主站` |
| URL | `https://acetoken.top` |
| 心跳间隔/重试次数/超时 | 保持默认即可 |

建议至少加：主站首页、`/api/status` 接口各一条。

## 三、创建状态页

1. 顶部「状态页面」→ 新建
2. 名称随意，**路径（slug）**只能填 `a-z 0-9 -`，例如 `acetoken`，记下来后面要用
3. 进入编辑页 → 「编辑状态页面」→ 添加分组 → 把监控项拖进分组
4. 点击右上角**保存**（最容易漏的一步）
5. 验证：浏览器打开 `http://服务器IP:端口/status/你的slug`，应显示监控卡片而非 404 或"无服务"

## 四、回填 New API

进入 **系统设置 → 仪表盘设置 → Uptime Kuma** → 添加分类：

| 字段 | 填什么 | 注意 |
|---|---|---|
| 分类名称 | 任意，如 `核心服务` | 仅展示用 |
| Uptime Kuma地址 | `http://服务器IP:端口` | **只到端口为止**，不要带 `/status/xxx` |
| 状态页面Slug | `acetoken` | 只填 slug 本身，不是完整链接 |

保存后回首页「服务可用性」卡片查看效果。若仍空白，先排查网络：New API 所在容器/服务器能否访问到 Uptime Kuma 地址（安全组、Docker 网络隔离等）。

## 五、进阶：真实模型连通性监控

普通 HTTP(s) 监控只能测"网站在不在线"，测不出"模型能不能调通"。要测真实调用链路（用户请求 → 网关 → 鉴权 → 计费 → 选渠道 → 上游模型），配置如下：

- 监控类型：`Keyword`（或 HTTP(s) 里配关键字校验）
- 方法：`POST`（默认 GET 务必改掉）
- URL：`https://acetoken.top/v1/chat/completions`
- 请求头：
  ```json
  {
    "Authorization": "Bearer sk-你的令牌",
    "Content-Type": "application/json"
  }
  ```
- 请求体：
  ```json
  {
    "model": "gpt-4o-mini",
    "messages": [{"role": "user", "content": "ping"}],
    "max_tokens": 1
  }
  ```
- 关键字：填成功响应必含的字段，如 `choices`

**必须注意：**

1. 这是真实调用，会真实消耗渠道额度/Token，`max_tokens` 设小一点，检测频率别太高（建议 5~10 分钟一次）
2. 用专门新建的监控令牌，不要用日常主令牌，并设额度上限
3. 一个监控项只能验证一个模型/一条链路，测不出具体是哪个渠道坏的——排查具体渠道故障仍需回**渠道管理**页看响应时间和状态
4. 想覆盖多个关键模型，就为每个模型单独建一条监控项

**推荐搭配：** 对外展示用 1~2 个真实模型调用监控（体现"能不能用"），内部排查具体渠道故障用 New API 自带的渠道测试功能。
